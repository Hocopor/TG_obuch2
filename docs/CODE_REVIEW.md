## 🔴 Критичные проблемы

### 1. Токен бота и пароли закоммичены в git
`.env` с реальным `TELEGRAM_BOT_TOKEN`, `CHAT_ID` и паролями находится в git-репозитории (коммиты `a88f7a6`, `2b6acf8`, `a497641`). `.gitignore` добавили позже — файл уже отслеживается и остаётся во всей истории.

Дополнительно: `ADMIN_PASSWORD=admin` и `DB_PASSWORD=postgres` — дефолтные слабые значения, при этом админка торчит наружу через `andrew-bot-adminka.mak-o.ru`.

**Что делать:**
1. Отозвать токен бота через @BotFather (`/revoke`) и выпустить новый.
2. `git rm --cached .env`, вычистить из истории (`git filter-repo`) — особенно если репозиторий куда-то пушится.
3. Сменить пароль админки и БД на стойкие.

### 2. Поиск пользователей в админке падает с ошибкой 500 ✔ проверено
`admin/app/routers/users.py:24` — `User.telegram_id.cast(str)`. SQLAlchemy не принимает Python-тип `str` в `cast()`:

```
TypeError: Object '' associated with '.type' attribute is not a TypeEngine class or object
```

Любой поиск (`/users?q=...`) — по имени, по нику, по чему угодно — возвращает 500. Это одна из ключевых функций админки по ТЗ.

**Фикс:** `from sqlalchemy import String` → `User.telegram_id.cast(String)`.

### 3. Загрузка юр-документов через админку не работает ✔ проверено
`admin/app/routers/legal.py:35` — параметр `document_type: str = ""` объявлен без `Form(...)`, поэтому FastAPI ждёт его в query-string, а форма (`legal.html`) шлёт его в теле POST. В обработчик всегда приходит `""`, `LegalDocTypeEnum("")` бросает `ValueError` → 500.

Воспроизведено на тестовом FastAPI-приложении: form-поле игнорируется, приходит пустая строка.

**Фикс:** `document_type: str = Form(...)`.

Это блокирует пункт ТЗ «сюда загружаются документы согласий и оферт» и пункт плана «Загрузка реальных юр-документов».

### 4. Загруженные документы сохраняются мимо Docker-volume
`legal.py:12`: `UPLOAD_DIR = Path(__file__).parent.parent.parent.parent / "uploads"`. Внутри контейнера файл лежит в `/app/app/routers/legal.py`, четыре `.parent` дают корень `/`, т.е. документы пишутся в **`/uploads`**. А volume в `docker-compose.yml` смонтирован в **`/app/uploads`**.

Итог: после пересборки/пересоздания контейнера все загруженные оферты и политики исчезнут, а ссылки в БД останутся битыми — пользователи при регистрации получат нерабочие ссылки на документы.

**Фикс:** задать `UPLOAD_DIR = Path("/app/uploads")` через env/константу, либо считать от `WORKDIR`.

### 5. Пользователь наглухо застревает в анкете «Предложить объект»
`bot/app/handlers/object.py` — в FSM-цепочке (название → адрес → описание → фото → видео → бюджет) **ни на одном шаге нет кнопок**. Хендлер `obj_cancel` существует, но ни одна клавиатура не содержит callback `obj_cancel` — это мёртвый код. Выйти из анкеты нельзя: любой текст трактуется как ответ на текущий шаг, `/start` не сбрасывает state (в `cmd_start` нет `state.clear()`), кнопки «Главное меню» на экране нет.

Единственный способ «выйти» — пройти все 6 шагов до конца.

### 6. «Отмена» в «Задать вопрос» не отменяет режим вопроса
Кнопка «Отмена» (`question_kb`) ведёт на callback `main_menu`, а `main_menu_callback` (`menu.py:54`) **не делает `state.clear()`**. Пользователь нажал «Отмена», увидел меню, но FSM-состояние `QuestionState.waiting_text` осталось активным — его следующее сообщение всё равно улетит как «вопрос» в поддержку.

## 🟠 Серьёзные баги

### 7. Отзыв согласия может упасть и не удалить данные (152-ФЗ)
`consent.py (revoke_confirm)` удаляет записи пользователя, но **не обнуляет `Object.assigned_to`** у чужих объектов, назначенных этому пользователю. Если админ назначал юзеру объект — при отзыве согласия будет нарушение внешнего ключа, транзакция откатится, данные останутся на сервере, а бот молча упадёт на этом апдейте. Бот при этом уже пообещал «данные будут удалены».

Также после удаления пользователю показывается главное меню, и он **продолжает пользоваться ботом без каких-либо согласий**: `ConsentMiddleware` пропускает всех, у кого нет записи в БД (`if not user: return await handler(...)`). Это противоречит ТЗ («без этого дальше не пропускает») и смыслу отзыва согласия.

Нюанс вдобавок: тема пользователя в группе поддержки (с его именем и перепиской) при удалении не трогается — персональные данные остаются в Telegram-группе.

### 8. Уведомление о назначении объекта: двойная и конфликтующая логика
Есть **два** механизма уведомления исполнителя:
- админка шлёт напрямую через `shared/notifier.py` (сырой HTTP-запрос к Telegram API);
- бот раз в минуту опрашивает БД (`mailing.py: check_new_orders`) и шлёт ещё раз.

При этом в `check_new_orders` фильтр `~Object.admin_notes.ilike('%[notified]%')` **не срабатывает при `admin_notes IS NULL`** (в SQL `NOT (NULL ILIKE ...)` = NULL → строка исключается). У всех объектов `admin_notes` по умолчанию NULL, поэтому бот-опрос фактически мёртв. «Случайно» это спасает от двойных уведомлений, но: (а) если у объекта появится заметка — юзер начнёт получать дубли каждую минуту до простановки `[notified]`; (б) плюс существует третий, вообще неиспользуемый модуль `bot/app/services/notification.py`.

**Рекомендация:** оставить один механизм (notifier из админки), выпилить `check_new_orders` и `notification.py`.

### 9. Часовой пояс рассылок: отправка уедет на 3 часа
Админ вводит время в `datetime-local` (локальное, для МСК — UTC+3), оно сохраняется как naive datetime, а бот сравнивает со `datetime.utcnow()` (`mailing.py:19`). Рассылка, запланированная на 15:00, уйдёт в 18:00 по Москве.

### 10. Рассылки: нет статуса «ошибка», нет восстановления, нет rate-limit
- Статус `error` из ТЗ никогда не выставляется: даже если все отправки упали, рассылка помечается `sent`.
- Если бот перезапустится посреди рассылки — она навсегда зависнет в `sending` и не возобновится.
- Отправка идёт без пауз: у Telegram лимит ~30 сообщений/сек, на большой базе будут flood-ошибки. Нужен `asyncio.sleep(0.05–0.1)` между отправками и обработка `RetryAfter`.

### 11. «Повторить рассылку» ломается на спецсимволах
`mailings.py:77` — `message_text` подставляется в redirect-URL без URL-кодирования. Текст с `&`, `=`, `#`, `+` обрежется или исказится в форме повтора. Нужен `urllib.parse.urlencode`.

### 12. Долгие `sleep` в хендлерах держат соединения с БД
Воронка после выбора цели спит 45+ секунд (5+20+20) прямо в хендлере, «Примеры работ» — столько же, «Отзывы» — 23 сек. Всё это время висит открытая сессия БД из `DbMiddleware` (pool_size=20, overflow=10). ~30 одновременных пользователей в воронке — и пул исчерпан, бот начнёт кидать `TimeoutError` на всех апдейтах. Правильно — планировать отложенные отправки через APScheduler (он создан в `main.py`, но не используется вообще) или отдельные задачи без удержания сессии.

### 13. Notifier шлёт с parse_mode=HTML и пользовательским текстом
`shared/notifier.py` подставляет название/описание объекта в HTML без экранирования. Объект с именем `квартира <центр>` → Telegram вернёт 400 → уведомление молча потеряется. Нужно `html.escape()` или убрать parse_mode.

### 14. Скачивание карточки объекта падает при пустой фамилии
`objects.py:127` — `obj.assigned_user.first_name + ' ' + obj.assigned_user.last_name`: если у назначенного юзера `last_name = None` (частый случай в TG) → `TypeError` → 500. Аналогично в шаблонах выводится «Иван None».

### Новые баги, найденные при втором проходе

- **Фото/голосовое вместо текста в анкете объекта роняет сохранение.** Шаги анкеты (`object.py`) не фильтруют по `F.text`: если на шаге «Введите название объекта» человек пришлёт фото, `message.text = None` запишется в state, а на финальном шаге `Object(object_name=None)` нарушит `nullable=False` → `IntegrityError`, вся анкета молча пропадёт — пользователь не получит ни подтверждения, ни ошибки.
- **Двойной клик по «Посмотрел»** запускает цепочку отзывов дважды (защиты от повторного нажатия нет; то же с кнопками «Отзывы», «Примеры работ» — каждая отправляет заново всю серию с задержками).
- **Мёртвый код, вводящий в заблуждение:** `welcome_kb` с callback `start`, у которого нет обработчика (и callback «start» внесён в белый список `ConsentMiddleware`); `tariffs_kb` с кнопками «Купить…» не подключена; `learn_tariffs`, `obj_cancel`, `services/notification.py` — не используются. При доработках легко править не ту клавиатуру/сервис.

### Третий проход: инфраструктура, рантайм, доки

**🔴 PostgreSQL и админка открыты всему интернету.**
В `docker-compose.yml` порты прокинуты без привязки к localhost: `"5433:5432"` и `"5000:5000"` слушают на 0.0.0.0. На публичном сервере это значит:
- PostgreSQL доступен снаружи по `сервер:5433` с логином `tg_bot` и паролем `postgres` — вся база (пользователи, согласия, переписка) открыта на подбор/подключение;
- админка доступна по голому HTTP `сервер:5000` **в обход Caddy и HTTPS** — пароль админа при логине летит открытым текстом.

`docs/ARCHITECTURE.md` при этом называет оба порта «внутренними» — это не так.
**Фикс:** `"127.0.0.1:5433:5432"` и `"127.0.0.1:5000:5000"` (Caddy на той же машине, ему хватит localhost).

**🟠 `callback.answer()` после долгих пауз гарантированно падает.**
В `examples.py` (`examples_callback`, `menu.py: menu_examples`) и `reviews.py` (`reviews_callback`) `await callback.answer()` вызывается **после** серии сообщений с задержками (45+ и 23 сек). Callback-query в Telegram живёт ~15 секунд — ответ на просроченный запрос даёт `TelegramBadRequest: query is too old`. Итог: у пользователя кнопка висит со спиннером до таймаута, в логах — исключение на каждый клик. В `goal_selected` сделано правильно (answer до задержек) — нужно так же везде.

**🟠 Т21. «Бесплатные уроки» нельзя задать внешней ссылкой.**
ТЗ: «Кнопка бесплатные материалы **переводит по внешней ссылке** на бесплатные уроки». В админке тип `free_lessons` можно только **загрузить файлом**, и бот выдаёт ссылку на скачивание этого файла с домена админки (`/legal/{id}/download`). Поля «внешний URL» (на GetCourse/YouTube) не существует. Уроки — это внешняя площадка, файлом их не загрузишь; функция в текущем виде нереализуема по назначению.

**🟠 FSM в памяти: рестарт бота молча ломает сценарии.**
`MemoryStorage` — при каждом деплое/перезапуске все состояния теряются: человек посреди анкеты объекта или ввода вопроса не получит никакой ошибки, а его следующее сообщение внезапно улетит в группу поддержки как «вопрос». Для продакшена нужен `RedisStorage` или хотя бы понимание этого поведения.

**Мелкие находки третьего прохода:**
- `docs/ARCHITECTURE.md` не соответствует коду: `config.py`/`database.py`/`models.py` описаны внутри `bot/app/`, реально лежат в `shared/`; портовая схема неверна (см. выше). Документация будет сбивать при доработках.
- Назначение объекта в админке синхронно ждёт два HTTP-вызова к Telegram (таймаут 15 сек каждый) — при недоступности Telegram запрос админки висит до 30 секунд.
- CSRF-токенов в формах админки нет — сейчас спасает только браузерный дефолт SameSite=Lax; стоит задать `samesite="lax"` у cookie явно, а не полагаться на дефолт.
- При отзыве согласия удаляется и `consent_log` — оператор теряет доказательство, что согласие было получено и отозвано (для споров по 152-ФЗ журнал согласий обычно хранят). Стоит обсудить с заказчиком.
- `ADMINKA_URL` по умолчанию `""` — если переменная не задана, бот отправит в Telegram относительные ссылки на документы, которые не работают. Лучше падать при старте с понятной ошибкой.
- Мусор в репозитории: `.mimocode/.cron-lock`, неиспользуемые медиа (`ждун.jpg`, `бабка на рынке.jpg`, `долбит в окно.mov`, `лёд.mov`, `труба с водой.mov`); `отзыв_анна.jpg` побайтово совпадает с `бабка на рынке.jpg`, `отзыв_владимир.jpg` — с `ждун.jpg` (заглушки — но легко забыть заменить, проверить перед запуском).
- Дублирующие мёртвые callback: `examples` (используется `menu_examples`), обработчик `enroll` дублирует `menu_enroll`.
- Плейсхолдер поиска в `users.html` обещает «Поиск по имени, username, ID» — телефон не упомянут и не ищется (см. А1).

### Четвёртый проход: тонкие места рантайма и добивка

**🟠 Фоновый цикл рассылок может молча умереть.**
`bot/app/main.py:71` — `asyncio.create_task(mailing_loop(bot))` без сохранения ссылки на таск. Event loop держит таски только слабыми ссылками; официальная документация asyncio прямо требует сохранять результат `create_task`, иначе сборщик мусора может уничтожить задачу на середине работы. Симптом будет самый неприятный из возможных: рассылки и уведомления просто перестают отправляться без единой строчки в логах, до рестарта. Фикс — сохранить таск в переменную уровня `main()`.

**🟠 `funnel_stage` — это не воронка, а «последнее действие».**
Поле перезаписывается назад любым кликом: пользователь, дошедший до `watched_lessons`, нажимает «Бесплатные материалы» — и его этап откатывается на `free_materials_viewed`; `/start` сбрасывает на `consent_done`. Аналитика воронки по этому полю (единственная в дашборде) недостоверна. Либо обновлять этап только «вперёд», либо строить воронку по `analytics_events` (события уже пишутся, но нигде не читаются).

**🟠 Юр-документы отдаются как принудительное скачивание.**
`legal.py:107` — `media_type="application/octet-stream"`: пользователь, кликнувший «Оферта» перед тем как дать согласие, получает не просмотр документа, а скачивание файла. На мобильном Telegram это несколько экранов действий — большинство не станет «ознакомляться», что подрывает саму юридическую конструкцию согласия. Отдавать PDF как `application/pdf` c `inline`, а лучше — HTML-страницей.

**Расхождение со STATE.md:** «админка полностью русифицирована» — в списке пользователей (`users.html`) цель и этап воронки выводятся сырыми enum-значениями (`own_objects`, `watched_lessons`), перевод есть только в карточке пользователя.

**Мелкие находки четвёртого прохода:**
- Все даты в админке показываются в UTC без конвертации — для админа в МСК всё «врёт» на −3 часа (регистрации, загрузки документов, история рассылок). Это шире, чем баг №9 про отправку.
- Ссылки на GetCourse захардкожены в `keyboards.py` — замена плейсхолдеров на боевые потребует правки кода и пересборки, хотя в проекте уже есть таблица `settings` и механизм динамических ссылок. Стоило вынести в настройки админки.
- Карточка пользователя: `mailing_logs` загружаются из БД (`session.refresh`), но в шаблоне не выводятся — история полученных рассылок админу не видна, лишний запрос впустую.
- `admin/app/static/style.css` и монтирование `StaticFiles` — мёртвые: ни один шаблон не подключает css, все стили инлайном в `base.html`.
- `ARCHITECTURE.md` описывает целые несуществующие модули: `anti_spam.py`, `services/user.py`, `admin/app/config.py`, `auth.py`, подпапки `templates/users|mailings|objects|legal` — документ написан до реализации и ни разу не сверялся с фактом. Иронично: там же `consent_log` описан как «журнал для юридической защиты», который код при отзыве согласия удаляет.
- `ConsentMiddleware`: команда в форме `/start@ИмяБота` не проходит белый список (`split()[0]` сравнивается с `/start` буквально) — существующий пользователь без согласий получит консент-экран вместо перезапуска.
- `get_or_create_user` без обработки гонки: два быстрых `/start` (даблклик по кнопке в клиенте) → два INSERT, второй упадёт на unique-констрейнте.
- Загрузка юр-документов не ограничивает ни тип, ни размер файла — можно залить что угодно любого объёма (диск, публичная выдача через `/legal/{id}/download`).
- Двойной сабмит формы «Создать рассылку» создаёт две одинаковые рассылки (нет защиты от повторной отправки формы).
- `uploads/` не добавлен в `.gitignore` — при локальном запуске загруженные документы попадут в коммит.

### Пятый проход: гонки, деградация во времени и мины под будущие шаги

**🟠 Закрытая или удалённая тема поддержки навсегда отрезает пользователя.**
Если админ закроет/удалит тему юзера в группе поддержки (естественное желание «прибраться»), все последующие сообщения этого пользователя будут падать на `send_message(message_thread_id=...)`, попадать в `except Exception` и превращаться в «Не удалось отправить. Попробуйте позже» — навсегда. `support_thread_id` при ошибке не сбрасывается и тема не пересоздаётся (`question.py:72,151,277`). Нужно: при ошибке «thread not found / topic closed» обнулять `support_thread_id` и создавать тему заново.

**🟠 Мина под следующий шаг плана: замена заглушек на реальный контент не подхватится.**
Кэш `cached_files` ключуется по пути файла. Когда по плану заменят `видео_пример_*.mp4` и картинки отзывов на реальные (пути те же — имена файлов зашиты в хендлерах), бот **продолжит рассылать старые заглушки** из кэша Telegram: путь совпал → отправка по старому `file_id`, до файла на диске дело не дойдёт. В STATE.md это знание есть («после DELETE FROM cached_files…»), но в плане Фазы 3 шага «очистить кэш» нет — наступят гарантированно. Правильный фикс — ключевать кэш по пути + mtime/хешу файла.

**🟠 Первый деплой: bot и admin одновременно выполняют `create_all` по одной БД.**
Оба сервиса стартуют после healthcheck БД и наперегонки создают таблицы/enum-типы. `checkfirst` не атомарен: оба видят «таблицы нет» → оба делают `CREATE` → один падает (duplicate table/type). `restart: always` замаскирует, но в логах будет мусор, а при невезении (упал на середине DDL) — половинчатая схема. Инициализировать БД должен один сервис (или отдельный init-шаг).

**🟠 Смена пароля БД сломает подключение, если в нём спецсимволы.**
`shared/config.py:12` подставляет `DB_PASSWORD` в DSN без экранирования. Я выше настоятельно рекомендую сменить пароль с `postgres` — так вот: пароль с `@`, `/`, `%` или `:` молча сломает разбор URL. Экранировать через `urllib.parse.quote_plus`.

**🟠 При исправлении бага №4 (uploads) не забыть про данные.**
В `legal_documents.file_path` уже сохраняются абсолютные пути вида `/uploads/...`. После починки каталога старые записи станут битыми (файлов по этим путям не будет) — скачивание вернёт 500. Нужна миграция путей или перезагрузка документов.

**Гонки (низкая вероятность, но ломают воронку):**
- Два сообщения подряд от нового пользователя обрабатываются параллельно → оба видят `support_thread_id = NULL` → создаются **две темы**, одна затирает другую, осиротевшая тема остаётся в группе, ответы админа в неё уходят в никуда.
- Холодный кэш file_id: два пользователя одновременно доходят до одного видео → оба загружают файл и делают select-then-insert в `cached_files` → у второго `IntegrityError` (unique по file_path) и обрыв воронки посреди отправки видео. Нужен upsert (`ON CONFLICT DO UPDATE`).

**Мелкие находки пятого прохода:**
- Поиск в админке по `@username` (как люди привыкли копировать) не находит никого — ник хранится без `@`, а `ilike` по шаблону `%@vasya%` не совпадёт. Срезать `@` из запроса.
- Любая команда, кроме `/start` (включая «/help», «/menu», опечатки), молча проглатывается: `auto_forward_text` отбрасывает всё, начинающееся с `/`, а других обработчиков нет — пользователь не получает вообще никакой реакции.
- Неконсентнутый пользователь получает **новый консент-экран на каждое сообщение** — напишет пять слов подряд → пять одинаковых простыней со ссылками.
- Статистика доставки рассылки нигде не видна: `mailing_logs` пишутся (sent/failed с текстом ошибки), но ни счётчиков в списке рассылок, ни детализации нет — админ видит «✅ Отправлено» даже если 100% отправок упало. Это заметно ослабляет пункт ТЗ «видеть статусы».
- Длинная рассылка блокирует уведомления о назначении объектов: `mailing_loop` последовательный, `check_new_orders` ждёт конца рассылки (на большой базе — десятки минут).
- Клики по кнопкам «Купить» (URL-кнопки) в Telegram принципиально неотслеживаемы — воронка в аналитике всегда будет обрываться на показе тарифов. Если заказчику нужен учёт переходов на GetCourse — нужен промежуточный редирект со счётчиком (например, через админку).
- Мёртвый код в `dependencies.py`: `SECRET_KEY` (генерируется и не используется), `get_current_user` (никем не вызывается); по всем файлам — пачка неиспользуемых импортов.
- `@app.on_event("startup")` — deprecated API FastAPI (предупреждение в логах; заменить на lifespan).

## 🟡 Замечания по качеству и безопасности

1. **Git-гигиена:** в репозиторий закоммичены `__pycache__/*.pyc` и исходные `.mov`-видео (десятки МБ) — `.gitignore` добавили после первого коммита, из индекса файлы не убраны (`git rm --cached`).
2. **Авторизация админки:** один глобальный токен в `app.state` — второй вход выбивает первого; сравнение пароля не constant-time; нет ограничения попыток входа; cookie без `Secure`-флага (за Caddy с HTTPS стоит добавить).
3. **Path traversal при загрузке документов:** `file.filename` используется как есть (`legal.py:49`) — имя вида `..\..\x` может записать файл вне каталога uploads; одинаковые имена файлов молча перезаписывают друг друга, при этом старые записи `LegalDocument` продолжают ссылаться на тот же путь с новым содержимым.
4. **Дублирование кода:** логика «создать тему в группе поддержки и переслать» скопирована трижды в `question.py` (текст-вопрос, автопересылка текста, автопересылка медиа) — при правке легко разъедутся.
5. **`get_or_create_user` не обновляет** username/имя при повторных заходах — данные в админке устаревают, поиск по новому нику не найдёт человека.
6. **`ConsentMiddleware`**: `event.message.chat` для CallbackQuery может быть None (старые сообщения) → AttributeError; плюс middleware открывает вторую сессию БД на каждый апдейт параллельно с `DbMiddleware`.
7. **Пересылка из поддержки** (`support.py`): голосовые, кружки, стикеры админов молча игнорируются — пользователь не получит ответ и никто этого не заметит; ошибки отправки глотаются `except: pass`.
8. ~~Прокси применяется только при рестарте бота — в UI об этом не сказано~~ — **снято**: в `settings.html` предупреждение о необходимости рестарта есть.
9. **Нет миграций (Alembic)** — только `create_all`; любое изменение моделей на проде придётся накатывать вручную (грабли с `ALTER TYPE` уже задокументированы в STATE.md).
10. **Нет пагинации** в списках пользователей/объектов/рассылок — при росте базы страницы станут неподъёмными.
11. **`shared/config.py`**: нет проверки, что `TELEGRAM_BOT_TOKEN`/`CHAT_ID` заданы — бот с пустым токеном упадёт с невнятной ошибкой глубоко в aiogram.
12. **Вопросы (`questions.status`)**: статус `answered` выводится в админке, но никогда не проставляется — все вопросы вечно «Ожидает».
